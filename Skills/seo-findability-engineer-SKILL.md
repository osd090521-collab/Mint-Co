# SEO Findability Engineer Skill

Use this skill when working on the Mint & Co website's SEO, search visibility, indexability, metadata, structured data, sitemap, robots.txt, local SEO signals, page architecture, content hierarchy, internal linking, Google Search Console setup, and search performance readiness.

This skill is for making the site easier to find, easier to understand, and more attractive to click from search.

This skill must never use spammy SEO tactics, fake claims, keyword stuffing, doorway pages, hidden text, misleading schema, or ranking guarantees.

---

## 1. Primary goal

The goal is maximum legitimate findability.

The Mint & Co website should be:

- Easy for Google to crawl
- Easy for Google to index
- Easy for users to understand
- Easy to click from search results
- Easy to navigate once landed
- Technically clean
- Locally relevant where appropriate
- Structured around real services and real user intent
- Fast, mobile-first, and trustworthy
- Built for long-term search credibility

SEO must support the core website job:

## Understand → Trust → Act

Search traffic is only useful if visitors quickly understand the offer, trust the business, and know what to do next.

---

## 2. SEO principles

Every SEO change must follow these principles:

- Help users first
- Help search engines understand the page second
- Make every important page crawlable
- Make every important page indexable
- Use specific, honest page titles
- Write useful meta descriptions
- Use clean heading structure
- Use accurate structured data
- Build internal links deliberately
- Avoid thin pages
- Avoid duplicate pages
- Avoid keyword stuffing
- Avoid fake local pages
- Avoid guaranteed ranking claims
- Avoid overpromising outcomes

Good SEO is clear, useful, technically sound, and trustworthy.

Bad SEO is spammy, bloated, fake, repetitive, and short-term.

- Unique title
- Unique meta description
- Canonical URL

- Open Graph description

- Twitter/X metadata where appropriate

Metadata should be:

- Specific
- Human-readable
- Click-worthy
- Accurate
- Brand-aligned
- Search-aware without being stuffed

Mint & Co | Clean, Premium Websites for Local Businesses
```

or:

```

## 9. Meta description rules

Meta descriptions should work like a clear search-result pitch.

They should:

- Explain the page
- Mention the service clearly
- Give a reason to click
- Stay honest
- Avoid stuffing
- Avoid hype
- Avoid unsupported claims

Good example:

```txt
Mint & Co builds clean, mobile-first websites that help local businesses look sharper, feel more trustworthy, and become easier to contact online.
```

Service page example:

```txt
Upgrade an old or missing website into a clean, mobile-first online presence with clear services, trust signals, contact routes, and Google-ready foundations.
```

Avoid:

```txt
The best SEO web design agency guaranteed to rank your business number one and generate unlimited bookings.
```

## 13. Canonical URL rules

Every indexable page should have a canonical URL.

Canonicals should:

- Use the production domain
- Point to the preferred page version
- Avoid query strings unless necessary
- Avoid duplicate trailing slash confusion
- Not all point to the homepage

Never canonical unrelated pages to one page just to hide duplication.

Fix duplication properly.

---

## 14. Sitemap rules

The sitemap should list all important public, indexable pages.

Include:

- Homepage
- Service pages
- Process page
- Work/case study pages
- About page
- Contact page
- Privacy page
- Terms page

Exclude:

- API routes
- Admin pages
- Test pages
- Draft pages
- Duplicate pages
- Internal preview pages
- Thank-you pages unless they have a real SEO purpose

If using Next.js, prefer a generated sitemap where possible.

Make sure the sitemap uses the production domain.

---

## 15. Robots.txt rules

Robots.txt should help crawlers understand what they can access.

Do not accidentally block the full site.

Dangerous rule:

```txt
Disallow: /
```

Only use this before launch if intentionally preventing crawling.

Robots.txt should usually include the sitemap location:

```txt
Sitemap: https://mintandco.co.uk/sitemap.xml
```

Remember:

- robots.txt controls crawling, not guaranteed indexing removal
- use noindex for pages that should not appear in search
- use password protection for truly private content

---

## 16. Indexing rules

Index pages that deserve search visibility.

Noindex pages that should not appear in search.

Usually index:

- Homepage
- Service pages
- About page
- Process page
- Case studies/work pages
- Contact page if useful

Usually noindex:

- Admin routes
- Internal tools
- Draft pages
- Test pages
- Search result pages
- Thin thank-you pages
- Duplicate landing pages
- Staging pages

Do not noindex important pages accidentally.

## 31. Redirect rules

Redirects should be clean and deliberate.

Use redirects when:

- A URL changes
- Pages are consolidated
- Old pages are replaced
- Trailing slash/domain versions need standardising
- HTTP redirects to HTTPS
- www redirects to non-www, or the chosen opposite

Use permanent redirects for stable URL moves.

Avoid chains like:

```txt
/page-a → /page-b → /page-c
```

Prefer:

```txt
/page-a → /page-c
```

## 38. SEO implementation order

When improving SEO, follow this order:

1. Inspect existing routes and pages
2. Identify which pages should be indexed
3. Identify which pages should not be indexed
4. Audit metadata
5. Audit headings
6. Audit crawlable internal links
7. Add or fix sitemap
8. Add or fix robots.txt
9. Add canonical URLs
10. Add Open Graph/Twitter metadata
11. Add structured data
12. Improve image alt text and filenames
13. Check mobile/performance issues
14. Check Search Console readiness
15. Confirm no spammy tactics or fake claims
16. Document what changed and what still needs manual setup

Do not start by adding random keywords.

## 41. Example sitemap pattern

Use this pattern where appropriate:

```ts
import type { MetadataRoute } from "next";

export default function sitemap(): MetadataRoute.Sitemap {
  const baseUrl = "https://mintandco.co.uk";

  return [
    {
      url: baseUrl,
      lastModified: new Date(),
      changeFrequency: "monthly",
      priority: 1,
    },
    {
      url: `${baseUrl}/services`,
      lastModified: new Date(),
      changeFrequency: "monthly",
      priority: 0.9,
    },
    {
      url: `${baseUrl}/process`,
      lastModified: new Date(),
      changeFrequency: "monthly",
      priority: 0.8,
    },
    {
      url: `${baseUrl}/contact`,
      lastModified: new Date(),
      changeFrequency: "monthly",
      priority: 0.8,
    },
  ];
}
```

Only include real existing routes.

## 42. Example robots pattern

Use this pattern where appropriate:

```ts
import type { MetadataRoute } from "next";

export default function robots(): MetadataRoute.Robots {
  const baseUrl = "https://mintandco.co.uk";

  return {
    rules: {
      userAgent: "*",
      allow: "/",
      disallow: ["/api/", "/admin/", "/drafts/"],
    },
    sitemap: `${baseUrl}/sitemap.xml`,
  };
}
```

Adapt disallowed routes to the real project.

Do not disallow routes that should rank.

## 43. Example JSON-LD injection pattern

Use JSON-LD carefully and accurately.

Example component:

```tsx
type JsonLdProps = {
  data: Record<string, unknown>;
};

export function JsonLd({ data }: JsonLdProps) {
  return (
    <script
      type="application/ld+json"
      dangerouslySetInnerHTML={{
        __html: JSON.stringify(data).replace(/</g, "\\u003c"),
      }}
    />
  );
}
```

Use this only with trusted schema objects generated by the app.

Do not put user-submitted content directly into JSON-LD without sanitisation.

## 46. Priority system

Use this priority system:

### P0 — Critical

Blocks crawling, indexing, deployment, or trust.

Examples:

- Site accidentally noindexed
- robots.txt blocks whole site
- No sitemap
- Broken canonical setup
- Important pages unreachable
- Production metadata missing
- Major broken links

### P1 — Important

Materially improves search clarity and click quality.

Examples:

- Weak title tags
- Duplicate meta descriptions
- Missing service page metadata
- Missing schema
- Poor heading structure
- Weak internal links
- Missing Search Console setup

### P2 — Useful

Good improvements but not urgent.

Examples:

- Better image filenames
- Expanded FAQs
- Improved Open Graph images
- Additional content clusters
- More case studies
- Better footer links
